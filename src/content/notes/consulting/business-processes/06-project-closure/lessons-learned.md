# Lessons Learned

## Overview

Capturing lessons learned is essential for continuous improvement and organizational learning. This document provides guidance on effectively capturing and utilizing lessons from Data and AI engagements.

---

## Why Lessons Learned Matter

- **Avoid Repeating Mistakes:** Learn from challenges
- **Replicate Success:** Scale effective practices
- **Improve Estimates:** Better planning accuracy
- **Develop Talent:** Share knowledge across team
- **Enhance Quality:** Continuous improvement

---

## When to Capture Lessons

| Timing | Focus |
|--------|-------|
| **During Project** | Real-time issues and solutions |
| **Phase Completion** | Phase-specific learnings |
| **Project Closure** | Comprehensive retrospective |
| **Post-Mortem** | Significant incidents |

## Lessons Learned Template

### Lesson Entry

| Field | Description |
|-------|-------------|
| **ID** | Unique identifier |
| **Category** | PM / Technical / Client / Team |
| **Phase** | When it occurred |
| **Title** | Brief descriptive title |
| **Situation** | What happened? Context |
| **Action** | What was done? Response |
| **Result** | What was the outcome? |
| **Lesson** | What did we learn? |
| **Recommendation** | What should future projects do? |
| **Applies To** | Project types / situations |

### Example Entries

**Lesson 1: Data Access Delays**
| Field | Content |
|-------|---------|
| ID | LL-001 |
| Category | Project Management |
| Title | Data Access Delays |
| Situation | Data access requests took 4 weeks instead of expected 2 weeks |
| Action | Escalated to client sponsor, adjusted project timeline |
| Result | 2-week delay to overall project |
| Lesson | Data access lead times are often underestimated |
| Recommendation | Request data access during pre-engagement; add buffer to timeline |

**Lesson 2: Effective Sprint Demos**
| Field | Content |
|-------|---------|
| ID | LL-002 |
| Category | Client Engagement |
| Title | Effective Sprint Demos |
| Situation | Initial demos were too technical, client stakeholders disengaged |
| Action | Restructured demos to focus on business outcomes first |
| Result | Increased client engagement and faster feedback |
| Lesson | Demo audience determines demo format |
| Recommendation | Tailor demos to audience; business outcomes before technical details |

## Conducting a Retrospective

### Preparation

1. Schedule session (1-2 hours)
2. Invite all team members
3. Gather project metrics
4. Review project timeline
5. Prepare facilitation materials

### Facilitation Techniques

**Start-Stop-Continue:**
- What should we START doing?
- What should we STOP doing?
- What should we CONTINUE doing?

**4 L's:**
- What did we LIKE?
- What did we LEARN?
- What did we LACK?
- What do we LONG for?

**Timeline Review:**
- Walk through project timeline
- Mark high points and low points
- Discuss significant events

### Ground Rules

- No blame - focus on learning
- All perspectives valued
- Confidentiality respected
- Specific examples encouraged
- Action-oriented outcomes

## Metrics to Review

At project close, review:

| Metric | Questions |
|--------|-----------|
| **Budget** | Was estimate accurate? Variance causes? |
| **Timeline** | On schedule? Delay causes? |
| **Scope** | Changes? Creep? Clarity issues? |
| **Quality** | Defects? Rework? Client satisfaction? |
| **Team** | Utilization? Satisfaction? Skill gaps? |
| **Risks** | Materialized risks? Unforeseen issues? |
